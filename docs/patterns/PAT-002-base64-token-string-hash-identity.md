패턴명: base64 토큰 문자열을 해시해 식별자로 쓰면 같은 사용자가 파편화된다
카테고리: auth
증상: HMAC 서명 토큰이 서명 검증은 통과하는데, 같은 사용자가 여러 개의 다른 식별자(owner)로 갈라져 자기 데이터를 못 찾는다
원인: base64(url) 인코딩은 바이트 수가 3의 배수가 아니면 마지막 문자에 미사용 비트가 남는다. 디코더가 그 비트를 무시하므로 같은 바이트열에 대해 유효한 문자열 표현이 여러 개 존재한다(16바이트 → 마지막 문자 4비트 미사용 → 표현 16가지). 서명 검증은 디코드된 바이트로 하고 식별자는 문자열 전체를 해시하면, 검증은 통과하면서 식별자만 달라진다
해결: 식별자를 문자열이 아니라 디코드된 바이트에서 파생한다(owner = SHA-256(rand bytes)). 또는 디코드 후 재인코딩한 값이 원본 문자열과 같은지 대조해 정규형(canonical form)을 강제한다
적용조건: 토큰·ID를 base64/base64url로 인코딩하고 그 문자열 자체를 해시·비교·키로 쓰는 모든 경우. 서명 검증의 입력과 식별자 파생의 입력이 다르면 포맷과 무관하게 발생한다

## 재현

SnapContext 0.4.0 토큰 형식은 `sc_<base64url(rand 16B)>.<base64url(HMAC 앞 16B)>` 다.
`verifyUserToken`은 두 조각을 디코드한 바이트로 HMAC을 재계산해 비교하고,
`ownerFromToken`은 `SHA-256(토큰 문자열 전문)`을 owner로 쓴다.

16바이트는 base64url 22자로 인코딩되는데 22×6 = 132비트라 4비트가 남는다.
마지막 문자를 그 4비트만 다른 값으로 바꿔도 디코드 결과는 동일하다.

```
원본: sc_TeGf5XF6Y-HtNyeuj805Mw.2Ab98B0W2WOwWVgrtv8Sow  → owner fde93314…
변형: sc_TeGf5XF6Y-HtNyeuj805Mw.2Ab98B0W2WOwWVgrtv8Sox  → owner 20338c3c…
```

두 문자열 모두 `verifyUserToken`을 통과한다. owner만 다르다.
body·sig 각각 16가지 표현이 있으므로 토큰 하나당 유효 변형은 255개.

## 영향 범위

권한 상승이나 타인 데이터 접근은 생기지 않는다. 변형을 만들려면 이미 유효한 토큰을
알아야 하고, 변형해봐야 비어 있는 새 네임스페이스가 생길 뿐이다. 실질 피해는
"자기 데이터를 못 찾음"(가용성)과 사용자당 행 파편화다.

다만 "같은 토큰 = 같은 사용자"라는 불변식이 문자열 정규형에 암묵적으로 기대고 있다는 점이
설계 결함이다. 클라이언트가 서버 응답을 그대로 저장하는 동안에만 성립한다.

## 교훈

서명을 **무엇으로 검증하는지**와 식별자를 **무엇에서 파생하는지**의 입력이 다르면
그 사이에 표현 다양성이 끼어든다. 둘 다 같은 정본(바이트열)에서 파생시켜라.

## SnapContext 적용 결과

`base64UrlDecode`가 디코드한 바이트를 재인코딩해 원본 문자열과 대조하고,
다르면 `null`을 반환하도록 고쳤다(`worker/src/token.ts`). 정규형이 아닌 표현은
`verifyUserToken` 단계에서 거부되므로 owner 파편화 자체가 불가능해진다.
식별자 정의(`SHA-256(토큰 전문)`)는 그대로 두고 입력 쪽을 좁힌 것이라
기존 발급 토큰과 저장된 owner 값은 전부 그대로 유효하다.

검증: `worker/test/token.test.ts`가 body·sig 끝 문자를 base64url 알파벳 전체로
치환해 통과하는 변형이 하나도 없음을 확인한다(수정 전에는 각각 15개가 통과했다).
