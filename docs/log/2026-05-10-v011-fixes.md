## 3. 이미지 확대 라이트박스

### 신규 컴포넌트
[`src/sidepanel/components/ImageLightbox.ts`](../../src/sidepanel/components/ImageLightbox.ts)

### 기능
- 미리보기 우하단 🔍 버튼 → 풀스크린 오버레이로 원본 이미지 표시
- 핀 베이크 안 된 **원본 이미지** 표시 (img.src 동일)
- ESC + backdrop 클릭 + 우상단 × 로 닫기
- **마우스 휠 줌** (0.5×~16×), **드래그 팬**
- 핀 추가/삭제 가능 (라이트박스 안에서 핀 클릭 = 즉시 삭제)
- 더블클릭 = 뷰 리셋

### 핵심 결정 (3차 시행착오 끝)
처음에 `transform: scale()` 으로 줌 구현 → GPU 컴포지팅 레이어가 GPU bilinear 보간으로 이미지를 보간 → CSS `image-rendering` 힌트가 무시되어 텍스트 ghosting 발생.

**최종 해결**: `transform: scale` 폐기, **img.style.width/height 직접 변경**으로 줌. Pan 도 transform: translate 대신 `viewport.scrollLeft/scrollTop` (네이티브 스크롤) 로. GPU 합성 우회 → `image-rendering: pixelated` 가 정확히 적용되어 휠 줌 시 또렷.

자세한 진단: [docs/troubleshooting/005-gpu-compositor-blur-on-transform-scale.md](../troubleshooting/005-gpu-compositor-blur-on-transform-scale.md)

## 4. Full Page Capture

### 메시지/타입
- [types/index.ts](../../src/types/index.ts) — `CAPTURE_FULL_PAGE`, `FULL_PAGE_CAPTURE_FAILED`, `FULL_PAGE_CAPTURE_TRUNCATED` 추가, `CaptureType` 에 `'full-page'` 추가
- [messaging.ts](../../src/utils/messaging.ts) — `FULL_PAGE_CAPTURE_STARTED` 응답 타입

### 백그라운드 구현
[service-worker.ts](../../src/background/service-worker.ts) `runFullPageCapture()`:

1. `measureFullPage()` — `chrome.scripting.executeScript` 로 `document.scrollHeight`, `viewportHeight`, `devicePixelRatio` 측정
2. `preparePageForFullPageCapture()` — `scrollBehavior: 'auto'` 설정, 원래 스타일 stash
3. 루프: 뷰포트 단위 스크롤 → fixed/sticky 요소 숨김 (첫 프레임 이후) → 250ms 안착 → `chrome.tabs.captureVisibleTab` → 청크 수집
   - 510ms 쓰로틀로 captureVisibleTab API ~2회/초 제한 회피
   - 마지막 청크는 잘라서 중복 영역 제거
4. `OffscreenCanvas` + `createImageBitmap` 으로 청크들 stitch
5. `restoreAfterFullPageCapture()` — 숨겼던 fixed/sticky 복원, 스크롤 0,0 으로
6. 결과를 `CAPTURE_RESULT` 로 broadcast (captureType: `'full-page'`)

### 제약
- 최대 높이 **15000px** (메모리 보호)
- 초과 시 `FULL_PAGE_CAPTURE_TRUNCATED` 메시지 → "페이지가 너무 길어 15000px까지 캡처했습니다" 토스트
- iframe 무시
- 일반 페이지 ~30-60초 소요

자세한 설계: [docs/adr/005-full-page-capture.md](../adr/005-full-page-capture.md)

## 6. 핀/메모 관련 버그 4종

### 6-1. 핀 메모 textarea 가 입력 안 됨

**증상**: 핀 추가 → 메모 textarea 가 나타나지만 클릭해도 입력이 안 됨.

**원인**: textarea 의 `focus` 이벤트 핸들러가 `onFocusPin → refreshPins()` 를 호출 → `memoList.render()` 가 `listRoot.innerHTML = ''` 로 textarea 를 destroy 후 재생성 → 포커스 소실 + 키 입력 누락. 매 포커스마다 무한 사이클.

**수정** ([App.ts](../../src/sidepanel/App.ts)): "활성 핀 변경" 과 "구조 변경" 분리.
- `onFocusPin`/`onSelectPin`: `pinLayerMain.render` + `memoList.highlightRow` 만 (textarea 보존)
- `refreshPins()` (풀 재렌더): 핀 add/delete 일 때만 호출

자세한 진단: [docs/troubleshooting/006-pin-memo-focus-loss-from-refresh.md](../troubleshooting/006-pin-memo-focus-loss-from-refresh.md)

### 6-2. 라이트박스 핀 클릭 시 새 핀 생성

**증상**: 라이트박스에서 기존 핀 위를 클릭하면 삭제되어야 하는데, 같은 자리에 새 핀이 추가됨.

**원인**: `viewport.setPointerCapture(pointerId)` 가 활성화된 상태에서 `pointerup` 이벤트의 `ev.target` 이 항상 viewport 가 됨 → `target.closest('.pin-badge')` 가 null 반환 → 삭제 분기 못 타고 추가 분기로.

**수정** ([ImageLightbox.ts:191-220](../../src/sidepanel/components/ImageLightbox.ts#L191-L220)): `ev.target` 대신 `document.elementFromPoint(ev.clientX, ev.clientY)` 로 실제 픽셀 위치 요소 조회.

### 6-3. "같은 핀 한 번 더 클릭하면 삭제" 토글 추가

**요구**: 핀 배지 첫 클릭 = 선택 (활성 강조), 같은 핀 두 번째 클릭 = 삭제.

**난점**: `onAddPin` 이 새 핀을 즉시 `activePinId` 로 설정 → 단순한 "isActive" 체크가 첫 클릭에 바로 삭제 분기로 빠짐.

**수정**: `lastClickedPinId` 별도 상태 도입.
- 사용자가 명시적으로 클릭한 핀만 추적 (auto-active 는 제외)
- `onAddPin` 시 `lastClickedPinId = null` 초기화
- `onSelectPin(id)` 에서 `lastClickedPinId === id` 면 삭제, 아니면 선택 + `lastClickedPinId = id`

[App.ts:140-180](../../src/sidepanel/App.ts#L140-L180), [PinAnnotation.ts](../../src/sidepanel/components/PinAnnotation.ts)

### 6-4. 히스토리에서 불러온 팩에 추가한 핀이 프롬프트에 안 들어감

**증상**: 히스토리 클릭 → 옛 팩 로드 → 새 핀 추가 + 메모 입력 → "AI 프롬프트 복사" → **`## 핀 주석` 섹션 통째 누락**.

**원인** ([ContextPackPanel.ts:311](../../src/sidepanel/components/ContextPackPanel.ts#L311)): `tryBuildPack()` 가 `if (loadedPack) return loadedPack` 로 short-circuit. 옛 annotations(빈 배열) 그대로 반환 → 템플릿의 `{{#if pins}}` 가 false → 섹션 누락.

**수정**: 라이브 캡처 입력이 있으면 항상 fresh 생성, `loadedPack` 은 캡처 상태 부재 시 폴백으로만.

자세한 진단: [docs/troubleshooting/007-loaded-pack-pin-prompt-mismatch.md](../troubleshooting/007-loaded-pack-pin-prompt-mismatch.md)

## 7. 캡처 메타 누락 — viewport 0×0, UA 공란

**증상**: YouTube 등 SPA 페이지 캡처 시 프롬프트의 "뷰포트: 0×0", "UA:" 공란.

**원인** ([service-worker.ts](../../src/background/service-worker.ts) `getPageMetaFromTab`): 콘텐츠 스크립트에 `GET_PAGE_META` 메시지 보내고 응답 없으면 `{viewport:{0,0}, userAgent:''}` 폴백. SPA 에서 콘텐츠 스크립트가 적시 등록 안 됨.

**수정**: `chrome.scripting.executeScript` 인라인 함수 폴백 추가 — 콘텐츠 스크립트 응답 실패 시 직접 `window.innerWidth/Height + navigator.userAgent` 조회.

## 12. 라이트박스 흐림 해결의 시행착오 4단계

처음 단순한 CSS 토글로 시작해서 렌더링 파이프라인 자체를 바꿔야 해결됐던 가장 길었던 이슈.

| 단계 | 시도 | 결과 |
|---|---|---|
| 1 | `image-rendering: pixelated` 라이트박스에만 적용 | 줌 시에는 또렷하지만 휠 한 번에 GPU 컴포지팅으로 쉽게 흐려짐 |
| 2 | `--zoomed` 클래스 토글 (scale > 1 일 때만 pixelated) | 매우 긴 캡처(16500px) fit 시 노이즈 패턴 (heavy downscale + nearest-neighbor) |
| 3 | 디스플레이 비율 기반 스마트 토글 (≥0.5 면 pixelated) | 토글은 정확하지만 transform: scale 자체가 GPU bilinear 강제 → 휠 한 번에 ghosting |
| 4 | **transform: scale 폐기, img.style.width 직접 변경** + native scroll pan | ✅ 해결. GPU 합성 우회, `image-rendering: pixelated` 정확히 적용 |

### 핵심 학습
- `transform: scale()` 이 적용되면 브라우저는 해당 요소를 GPU 컴포지팅 레이어로 승격
- GPU 컴포지팅에서는 **GPU bilinear 보간**이 사용되며, CSS `image-rendering` 힌트가 무시됨
- 따라서 줌은 transform: scale 가 아니라 **요소 크기 직접 변경** 으로 해야 image-rendering 힌트가 정확히 작동

자세한 ADR: [docs/adr/007-lightbox-width-based-zoom.md](../adr/007-lightbox-width-based-zoom.md)
