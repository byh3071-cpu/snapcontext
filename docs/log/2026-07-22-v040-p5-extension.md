# 0.4.0 P5 — 확장이 worker 신기능을 실제로 쓰게

## 왜

P1~P4 로 worker 는 준비됐다. 토큰 발급 엔드포인트가 있고, 업로드에 토큰을 실으면 owner 가 찍히고, 보관 기간을 1/7/30 으로 고를 수 있고, MCP 는 사용자별로 격리된다.

**그런데 확장이 그걸 아무것도 안 썼다.** `src/utils/upload.ts` 는 `image` 와 `context` 만 보냈다 — 토큰도 보관 기간도 없었다. 서버에 문이 생겼는데 확장이 문을 두드릴 줄 모르는 상태였다. P5 가 그 배선이다.

## 설계 판단 3개 (되돌리려는 사람을 위해)

### 토큰 발급은 서비스워커가 아니라 사이드패널에서, 그것도 업로드 직전에

`chrome.runtime.onInstalled` 를 쓰지 않았다. 이유가 셋이다.

1. 확장 전체에 `onInstalled`/`onStartup` 핸들러가 **0개**였다. 없던 표면을 새로 여는 것보다 기존 패턴을 따르는 게 낫다.
2. e2e 의 fetch mock 은 `page.evaluate` 로 **사이드패널 페이지의 `window.fetch` 에만** 설치된다. 서비스워커에서 발급하면 mock 을 우회해 실제 네트워크로 나간다.
3. 사이드패널 초기화 시점도 안 된다 — e2e 가 `page.goto` → 대기 → `installFetchMock` 순서라, init 에서 발사하면 **mock 설치 이전**에 나간다.

그래서 `onShare` 안, 동의 통과 후 `uploadShare` 바로 앞에서 부른다. 기존 "최초 1회 동의"(storage 플래그 + lazy)와 같은 모양이다.

### `uploadShare` 는 chrome API 에 묶지 않는다

토큰을 함수 안에서 가져오게 하면, 지금 chrome mock 없이 도는 기존 테스트 6건이 전부 mock 을 요구하게 된다. 호출측이 토큰을 주입하는 순수 함수로 유지했다. 결과적으로 **기존 6건이 import 한 줄만 바뀌고 본문 0줄 변경으로 그대로 그린**이다.

### 부재와 빈 값을 구분한다

worker 계약이 둘을 다르게 처리한다.

- `Authorization` 을 붙였는데 토큰이 무효 → **401**. 토큰이 없으면 **헤더 키 자체를 만들지 않는다**(빈 문자열도 `Bearer ` 도 아니다).
- `expiresInDays` 가 빈 문자열 → **400**. 미지정이면 필드를 **append 하지 않는다**.

이 두 가지를 각각 "헤더 키가 없음"·"필드가 없음" 단언으로 못 박았다. 뮤테이션으로 확인했다 — falsy 가드를 제거하면 red 가 난다.

## in-flight 가드

사이드패널이 열릴 때와 업로드 직전에 동시에 부를 수 있다. 모듈 스코프에 Promise 를 캐시해 **동시 호출이 발급 요청을 2번 보내지 않게** 했다. worker 에 분당 10회 rate-limit 이 있고, 중복 발급은 owner 파편화로 이어진다.

e2e 가 이걸 브라우저 컨텍스트에서 검증한다 — 업로드를 여러 번 해도 `/token` 요청은 1회다.

## 만료 문구 동적화

"7일"이 `ImageActions.ts` 7곳에 박혀 있었다. 그중 5곳(섹션 aside·발행 캡션·버튼 title·버튼 라벨·`finally` 복원)은 **마운트 시점에 값이 정해지고 갱신 훅이 없었다.**

단일 `renderExpiryTexts()` 로 수렴시키고, 설정 변경은 `window` CustomEvent 로 전파한다(`storage/history.ts` → `HistoryList.ts` 의 기존 선례를 따랐다).

**동의 문구가 특히 중요하다.** "7일 후 삭제됩니다"라고 동의받고 30일로 저장하면 사실과 다른 동의가 된다. `onShare` 진입 시 기간을 한 번 고정해서 동의·전송·성공 토스트가 **같은 값**을 쓰게 했다.

성공 토스트의 만료일은 서버 응답에서 알 수 없다 — `/upload` 응답은 `{id, url}` 뿐이고 ADR-013 이 `expiresAt` 를 의도적으로 뺐다. 로컬 선택값으로 계산한다.

## e2e 하네스 수정

`installFetchMock` 이 `/upload` 만 가로채서 **`POST /token` 이 실제 워커로 나가고 있었다.** 지금 `TOKEN_SIGNING_SECRET` 미주입이라 매번 500 을 받고, 토큰 왕복 지연이 업로드 대기와 겹치면 기존 단언이 오탐으로 실패할 수 있었다. 테스트가 네트워크·시크릿 주입 상태에 의존하면 안 된다.

`/token` 도 가로채면서 P5 계약 3건을 추가로 검증한다 — Bearer 토큰 동봉, 보관 기간 전송, **토큰 발급 1회**.

### 시크릿 주입 후 1회 확인해야 할 것

**`POST /token` 의 Origin 검증**은 자동 테스트로 덮이지 않는다. worker 는 `Origin` 이 `chrome-extension://` 로 시작하지 않으면 403 인데, e2e 가 `/token` 을 mock 하면서 이 계약을 검증하는 경로가 사라졌다(mock 도입 자체는 네트워크 의존을 없애려는 올바른 선택이다).

실패해도 증상이 **"조용히 전원 익명"** 이라 아무 테스트도 red 가 안 된다. 실 워커로 스모크 1회가 필요하다.
