## 왜 — 상수 하나에 매달린 만료

기존 만료는 `worker/src/lib.ts` 의 `MAX_AGE_MS = 7일` 상수 하나가 전부였다. 문제는 판정 소스가 두 갈래였다는 것:

| 소스 | 쓰는 곳 |
|---|---|
| R2 `uploaded` + 7일 | `/i/` · `/s/` · `pack.ts` 이미지 head · `pack.ts` `{id}.json` |
| D1 `expires_at` | `snap_history` |

기간이 1/7/30 으로 갈라지는 순간 두 소스는 즉시 모순난다(D1 은 30일로 인덱싱했는데 R2 판정은 7일에 410). 그래서 이번 작업의 절반은 기능 추가가 아니라 **"판정과 표시가 같은 값을 보게 만드는 것"** 이었다.

## 설계 축 3개

1. **`readExpiry(obj) → ExpiryInfo`** 가 절대 만료시각·보관일수·출처(`metadata`/`legacy`/`invalid`)를 한 번에 반환한다. `formatExpiryKST`·`buildViewerHtml` 도 이 구조체를 받게 바꿔서, 라벨과 일수를 따로 넘기다 생기는 드리프트(라벨은 30일치인데 문구는 "7일")를 **타입으로 차단**했다.
2. **이원화 봉인** — 저장소를 합치지 않았다. `/upload` 가 `expiresAtIso` 를 1회 계산해 `expiryMeta` 객체 **같은 참조**를 두 put 에 넘기고, 같은 문자열을 D1 insert 에도 넘긴다. `captureRowFromSharedContext` 를 옵션 객체로 바꿔 `expiresAtIso` 를 필수 필드로 만들어서, 누락이 컴파일 에러가 된다.
3. **살아있는 표면만 동적화** — 뷰어 notice 는 `${retentionDays}일` 로 동적, 410 문구는 일수를 뺐다.

## 판단 기록 (되돌리려는 사람을 위해)

- **파싱 실패 = 만료 처리.** `customMetadata.expiresAt` 가 깨졌을 때 조용히 7일로 되돌리면 1일 캡처가 7일 산다(과보관). `source: 'invalid'` + 즉시 만료 + `console.warn` 으로 드러낸다.
- **`parseExpiresInDays` 는 정규식 `^\d+$` 를 먼저 통과시킨다.** `Number()` 만 쓰면 `'0x7'`·`'7e0'`·`' 7 '`·`'7.0'`·`'+7'` 이 전부 7 로 통과한다(테스트로 못박음). **빈 문자열은 400** — "부재=7" 은 필드가 없을 때의 규칙이지 빈 값의 규칙이 아니다.
- **`{id}.json` 에도 메타를 심는다.** `pack.ts` 가 이미지 head 와 `{id}.json` 을 **각각** 판정하므로, 이미지에만 심으면 30일 캡처가 8일째에 `snap_pack`·`snap_analyze` 만 죽고 `/s/`·`/i/` 는 사는 split-brain 이 된다. 이 fail-closed 계약은 `test/expiry.test.ts` 에 고정했다.
- **`/i/` 200 의 `Cache-Control` 에 `Math.max` 보정을 넣지 않았다.** 이 지점은 `isExpiredAt` 가드 통과 후라 잔여초가 음수일 수 없다. 보정을 넣으면 결함이 조용히 덮인다.
- **`buildExpiredHtml()` 은 인자 0개를 유지한다.** 물리 삭제된 객체는 보관일수를 알 방법이 없고, 아는 경우에만 일수를 붙이면 "이 id 는 실재했다"는 존재 오라클이 응답 차이로 새어 나간다.
- **D1 마이그레이션은 추가하지 않았다.** `expires_at` 절대시각으로 충분하다 — days 컬럼도, 인덱스도 불필요.
- **`scripts/lib/backfill-d1.mjs` 의 자체 7일 상수는 그대로 뒀다.** 백필 대상은 정의상 메타 없는 레거시 객체라 `readExpiry` 의 legacy 분기와 같은 계산이다. 바꾸면 오히려 어긋난다.

## 제약 — R2 쓰기는 Workers 바인딩 경유만

R2 를 **S3 호환 API** 로 쓰면 커스텀 메타데이터 키가 소문자화된다(`expiresAt` → `expiresat`). `readExpiry` 는 `customMetadata.expiresAt` 를 캐멀케이스로 읽으므로, S3 API 로 쓴 객체는 메타가 없는 것으로 보여 **레거시 7일 경로로 떨어진다**(30일 캡처가 7일에 만료). 따라서 이 프로젝트에서 **R2 쓰기는 Workers 바인딩(`env.BUCKET.put`) 경유만** 허용한다. 백필·마이그레이션 스크립트를 S3 SDK 로 짜지 말 것.

## 남은 운영 게이트 (사람) — 배포 선행 조건

**R2 버킷 lifecycle `auto-delete-7d` 를 30일로 상향하는 것이 배포의 선행 조건이다.** 순서를 지키지 않으면:

1. 30일 캡처가 **7일째 물리 삭제**된다(코드 판정은 30일인데 객체가 사라짐).
2. 더 나쁜 건 캐시다 — `/i/` 가 `max-age` 를 최대 30일치로 내보낸 뒤 객체가 7일에 사라지면, 클라이언트 캐시가 **최대 30일간 유령 서빙**한다. 서버는 이미 410 인데 브라우저는 옛 이미지를 계속 보여준다.

`/upload` 는 **공개 엔드포인트**라 배포 즉시 누구나 `expiresInDays=30` 을 보낼 수 있다. 따라서 lifecycle 상향과 배포는 **같은 창에서** 처리해야 한다(lifecycle 먼저 → 배포).

```shell
wrangler r2 bucket lifecycle list snapcontext-uploads   # 현재 규칙 확인 (wrangler.jsonc 의 bucket_name)
# auto-delete-7d 를 30일로 상향한 뒤 deploy
```

그 외 배포 전 확인:

- 상향 후 `lifecycle list` 재조회로 규칙이 실제 반영됐는지 확인(적용 지연 주의).
- 1일 캡처의 물리 삭제는 lifecycle 이 30일이어도 **코드 레벨 410 으로 차단**되므로 기능상 문제 없다(과보관 = 스토리지 비용 이슈일 뿐). 최소 보관창을 1일로 좁히는 건 별도 규칙이 필요하고, 이번 스코프 아님.

### 적대 검증에서 나온 관찰 (전부 non-blocker · 현재 도달 불가)

머지 전 적대 검증(claude critic)이 40여 종의 우회 입력을 실행해봤고 뚫리는 경로는 없었다. 다만 방어심도 관점의 관찰 4건을 기록해둔다 — 지금 고치지 않는 이유까지 남긴다.

- **`readExpiry` 는 파싱만 되면 임의 미래시각을 무제한 신뢰한다.** `expiresAt = '+275760-09-13T00:00:00.000Z'` 면 사실상 영구 미만료 + `max-age=8640000000000`. 유일한 writer 인 `/upload` 가 서버 계산값만 쓰고 R2 직접 쓰기는 자격증명이 필요해 **현재 도달 불가**다. 상한을 걸면 "메타가 SoT" 설계와 상충하므로 보류했다. 반대 방향(과거시각)은 fail-closed 로 떨어진다.
- **D1 경로와 R2 경로가 만료 정각 1ms 에서 반대로 판정한다.** R2 는 `expiresAtMs < now`(정각 유효), D1 은 SQL `expires_at > nowIso`(정각 제외). 부등호 등호 처리 차이로 **기존부터 있던 semantics** 이지만, 이제 두 값이 같은 문자열이라 경계가 실제로 도달 가능해졌다. 영향은 1ms 창.
- **`/s/` 는 `{id}.json` 의 만료를 검사하지 않는다**(이미지 head 만). `pack.ts` 는 양쪽을 보므로 `/s/` 가 `snap_pack` 보다 느슨하다. `/upload` 가 항상 양쪽에 메타를 심으므로 도달 불가지만, "R2 판정은 전부 fail-closed" 라고 읽으면 오해다.
- **`buildExpiredHtml.length === 0` 계약 테스트는 기본값 인자를 못 막는다.** `days = 7` 같은 기본값 인자는 `Function.length` 가 0 이라 통과한다. 존재 오라클 방지 의도를 완전히 덮으려면 `expect(buildExpiredHtml()).not.toMatch(/\d+일/)` 를 함께 둬야 한다.
- `/upload` 응답에는 `expiresAt` 를 **넣지 않았다** — 소비자가 없고(확장은 P5), 응답 키 화이트리스트 테스트(`{id,url}` 만)가 누출 방지 앵커로 남아 있다.
